# External Audit

## ML Core-Modeling Opportunity Improvements

Current state in this repository is a hybrid of high-quality domain heuristics and light ML-style modules, but without a formal learning/evaluation loop. The largest opportunity is not "more AI prompts"; it is tightening the modeling lifecycle: objective definition -> ground-truth capture -> leakage-safe training/evaluation -> calibrated inference -> monitored feedback.

1. **Unify the canonical inference path to remove train/serve skew (very high impact, medium effort).**  
   - **What to improve:** Consolidate modeling logic so production recommendations are generated by one canonical feature + scoring path (preferably server-side), rather than two partially divergent stacks.  
   - **Why it matters:** The app-side engine (`trainingAnalysis` + `workoutEngine`) is significantly richer than backend `engines/ml/*`; this creates inconsistent behavior depending on execution path and makes offline evaluation invalid.  
   - **Evidence:** `app/src/lib/trainingAnalysis.ts`, `app/src/lib/workoutEngine.ts`, `backend/src/engines/ml/readiness.js`, `backend/src/engines/ml/prediction.js`, `backend/src/routes/ml.js`, `backend/src/database/index.js`.  
   - **Implementation direction:** Define a shared feature contract and single inference entrypoint, then add parity tests that assert equivalent outputs for the same input snapshot.

2. **Replace static readiness heuristics with calibrated risk prediction (very high impact, medium effort).**  
   - **What to improve:** Move from fixed weighted sums and bins to a probabilistic model for short-horizon risk (under-recovery, overreach, likely underperformance).  
   - **Why it matters:** Current thresholds are global and hand-tuned, which will misclassify users with atypical baselines (especially HRV/RHR variability).  
   - **Evidence:** `backend/src/engines/ml/readiness.js` (hardcoded 0.3/0.3/0.2/0.2 weights and static red/yellow/green cutoffs).  
   - **Implementation direction:** Train a calibrated classifier/regressor using lagged health/training signals; report probability + confidence interval, not just zone.

3. **Upgrade prediction rigor from heuristic trending to leakage-safe forecasting (high impact, medium effort).**  
   - **What to improve:** Replace short-window average/split heuristics with time-aware forecasting and explicit uncertainty estimates.  
   - **Why it matters:** Current "trend" and "consistency" logic is easy to destabilize with sparse or noisy sessions; confidence labels are not statistically grounded.  
   - **Evidence:** `backend/src/engines/ml/prediction.js` (`last 20 workouts`, first-half vs second-half trend, string confidence).  
   - **Implementation direction:** Use rolling-origin validation, optimize MAE/MAPE (for volume/intensity) and AUROC/PR (risk labels), and output calibrated numeric confidence.

4. **Prevent label leakage and model self-reinforcement (very high impact, low-medium effort).**  
   - **What to improve:** Partition feedback into source types and block model-generated opinions from becoming direct training targets.  
   - **Why it matters:** Feeding LLM pattern observations back into profile features can produce circular reinforcement rather than learning from user outcomes.  
   - **Evidence:** `app/src/lib/workoutEngine.ts` (`parseLlmPatternObservations`), `app/src/lib/trainingAnalysis.ts` (`llmPatternObservations` in profile assembly), `app/src/pages/TodayWorkout.tsx`, `sql/migration_ml_v2.sql` (`model_feedback`).  
   - **Implementation direction:** Add provenance columns (`human`, `model_review`, `system_rule`) and train only on verified outcome labels.

5. **Define first-class ground-truth labels beyond coarse compliance proxies (very high impact, medium effort).**  
   - **What to improve:** Capture set-level execution quality and delayed outcomes (adherence quality, fatigue, pain/injury flags, next-day readiness impact).  
   - **Why it matters:** Current signal quality is dominated by coarse aggregates; objective misspecification will cap model quality regardless of algorithm choice.  
   - **Evidence:** `app/src/lib/trainingAnalysis.ts` (`computePrescribedVsActual`), `sql/migration_ml_v2.sql` (`generated_workouts`, `exercise_swaps`, `model_feedback`).  
   - **Implementation direction:** Add event tables keyed by `generated_workout_id` for per-set target-vs-actual and post-session outcomes.

6. **Introduce delayed label materialization windows (high impact, medium effort).**  
   - **What to improve:** Materialize labels at explicit horizons (D+1, D+7, D+28) with censorship rules for incomplete outcomes.  
   - **Why it matters:** Many fitness outcomes are delayed; immediate labels bias training toward short-term effects and can punish correct long-term prescriptions.  
   - **Evidence:** `app/src/lib/trainingAnalysis.ts` uses immediate/near-immediate profile-derived feedback; no explicit horizon materialization pipeline in backend.  
   - **Implementation direction:** Build async label jobs that only produce training rows when horizon windows close.

7. **Add immutable inference traceability and dataset versioning (high impact, low-medium effort).**  
   - **What to improve:** Persist model/config/feature version identifiers with each recommendation and outcome row.  
   - **Why it matters:** Without reproducibility, you cannot perform reliable regression analysis, rollback, or policy attribution.  
   - **Evidence:** `app/src/lib/modelConfig.ts` is used but not stamped into generated inference records; persistence schema in `sql/migration_ml_v2.sql` lacks full version metadata.  
   - **Implementation direction:** Store `engine_version`, `feature_schema_version`, `feature_snapshot_id`, `config_version`, and `inference_id`.

8. **Build a formal offline evaluation harness and benchmark suite (high impact, medium effort).**  
   - **What to improve:** Create deterministic evaluation jobs with leakage-safe splits and cohort segmentation (beginner/intermediate/advanced, recovery profiles, goal types).  
   - **Why it matters:** Improvements are currently hard to prove; shipping changes without benchmark gates increases regression risk.  
   - **Evidence:** No dedicated evaluation framework surfaced in `backend/src/engines/ml/*` and the app-side engine files; confidence and trend quality are mostly heuristic.  
   - **Implementation direction:** Add reproducible eval scripts and minimum acceptance thresholds per objective.

9. **Instrument online model-quality metrics and feedback attribution (high impact, low-medium effort).**  
   - **What to improve:** Track model outcomes server-side (acceptance, completion quality, progression delta, intervention regret).  
   - **Why it matters:** Current analytics queue is localStorage-backed and not a durable learning signal; online learning loops remain blind.  
   - **Evidence:** `app/src/utils/analytics.ts` stores events in localStorage only; `backend/src/utils/logger.js` lacks production telemetry sink.  
   - **Implementation direction:** Emit structured events to backend ingestion with schema versioning and quality checks.

10. **Implement drift and data-quality monitoring with alert thresholds (high impact, medium effort).**  
   - **What to improve:** Monitor feature distribution shift, missingness spikes, schema drift, and label delay rates.  
   - **Why it matters:** Fitness and wearable data are non-stationary; unmonitored drift silently degrades recommendation quality and trust.  
   - **Evidence:** No dedicated drift monitor in `backend/src/engines/ml/*`; logging currently console-first in `backend/src/utils/logger.js`.  
   - **Implementation direction:** Add daily monitoring jobs with alerting for distribution shift and data freshness SLAs.

11. **Strengthen confidence semantics and safe-fallback policy (medium-high impact, low effort).**  
   - **What to improve:** Replace categorical confidence (`high`/`medium`) with calibrated probabilities and explicit low-confidence behavior.  
   - **Why it matters:** Confidence fields currently influence decisions but are not tied to empirical reliability; this can overstate certainty.  
   - **Evidence:** `backend/src/engines/ml/prediction.js` confidence logic and several heuristic branches in `app/src/lib/workoutEngine.ts`.  
   - **Implementation direction:** Add calibration layer and policy guardrail: if confidence < threshold, use conservative deterministic fallback.

12. **Harden data contracts between ingestion and modeling (medium impact, low-medium effort).**  
   - **What to improve:** Enforce strict typed contracts for workout payload shape required by model code (nested exercises/sets completeness, naming conventions).  
   - **Why it matters:** Silent null/default handling can produce valid-looking but low-fidelity features, contaminating training/evaluation.  
   - **Evidence:** `backend/src/database/index.js` returns broad `select('*')` rows; modeling code expects nested structures in `backend/src/engines/ml/prediction.js`; ingestion flow in `backend/src/routes/input.js`.  
   - **Implementation direction:** Add schema validation at boundaries and reject/flag nonconforming records before they enter feature computation.
